# GLOBALO — Growth Engine Architecture

## 1. SEO OPTIMIZATION

### 1.1 Dynamic meta tags
- **Service**: `App\Services\SeoService` — builds `meta description`, `title`, `keywords` per page type.
- **Controller**: Each controller can pass `$seo = SeoService::forPage($type, $data)` to the view.
- **Layout**: `layout_desktop.php` outputs `$seo['title']`, `$seo['description']`, `$seo['canonical']` when set.

### 1.2 SEO-friendly URLs
- **Current**: `/experts`, `/experts/show/123` (expert profile).
- **Recommended**: Keep clean segments; optional slug in URL (e.g. `/experts/123/developer-john`). Slug can be added later without breaking existing links.

### 1.3 Structured data (Schema.org)
- **Organization**: on homepage.
- **Person** + **Service**: on expert profile pages.
- **BreadcrumbList**: on listing and detail pages.
- Snippets injected in layout via `$seo['structured_data']` (JSON-LD).

### 1.4 OpenGraph & Twitter Cards
- Set in layout from `$seo['og_*']` and `$seo['twitter_*']`.
- Expert profile: OG image = expert avatar or platform default, title = expert title, description = bio excerpt.

### 1.5 Sitemap & robots
- **sitemap.xml**: Generated by `SeoController::sitemap()` — homepage, /experts, /experts/show/{id}, /home/apropos, /home/contact, /blog (when blog exists). Updated dynamically from DB.
- **robots.txt**: Served by `SeoController::robots()` — Allow all, Sitemap: {BASE_URL}/sitemap.xml, disallow /admin, /client, /expert, /auth (private areas).

### 1.6 Canonical URLs
- Every page sets `<link rel="canonical" href="...">` to the current URL (no duplicate content).

### 1.7 Page speed
- Preconnect for fonts and tracking domains.
- Defer non-critical JS (e.g. growth.js, chatbot).
- Optional: image lazy-loading, minified CSS/JS (build step).

## 2. TRACKING AND ANALYTICS

### 2.1 Configuration (Admin or .env)
- `GA_MEASUREMENT_ID` (e.g. G-XXXXXXXXXX)
- `FB_PIXEL_ID`
- `LINKEDIN_PARTNER_ID` (optional)

### 2.2 Loading strategy
- **growth.js**: Loaded after cookie consent. If user accepts “analytics”, inject GA gtag.js and Facebook Pixel. If “marketing”, enable Pixel advanced matching and LinkedIn Insight Tag.
- **Event tracking**: `window.growthTrack(eventName, params)` — defined in growth.js, sends to GA (and optionally Pixel) only when consent given.

### 2.3 Events tracked
| Event name           | When                         | Params (example)                    |
|----------------------|------------------------------|-------------------------------------|
| sign_up              | User registers               | method, role                        |
| view_expert_profile  | Expert profile page view     | expert_id, competence               |
| begin_booking        | Client starts reservation    | expert_id, demande_id               |
| purchase             | Payment completed            | value, currency, reservation_id     |
| session_complete     | Session marked done          | reservation_id, expert_id           |

Conversion tracking: `purchase` and `sign_up` mapped to GA4 conversions and Facebook Pixel “CompleteRegistration” / “Purchase”.

## 3. COOKIE CONSENT (GDPR)

### 3.1 Banner
- Shown on first visit (no consent cookie).
- Options: “Accept all”, “Reject non-essential”, “Customize” (analytics on/off, marketing on/off).
- Consent stored in cookie `globalo_consent` (JSON: `{ analytics, marketing, timestamp }`) and in `localStorage` for JS.

### 3.2 Behavior
- **No consent**: Only strictly necessary (e.g. session, CSRF). No GA, no Pixel.
- **Analytics**: GA and basic event tracking.
- **Marketing**: Facebook Pixel, LinkedIn Insight, retargeting scripts.
- User can revoke later via “Cookie preferences” in footer; banner or modal to change choices.

### 3.3 Implementation
- `Views/partials/cookie_banner.php` + `public/assets/js/cookie-consent.js`.
- Layout includes banner and script; growth.js checks consent before loading third-party scripts.

## 4. SOCIAL MEDIA RETARGETING

- **Facebook / Instagram**: Same Pixel; audiences built in Meta Events Manager (visitors, non-converters). Custom audience “Website visitors” + “Excluded: signed up”.
- **LinkedIn**: Insight Tag on site; retargeting in LinkedIn Campaign Manager.
- **Ad copy example**: “Need help with coding or design? Find an expert instantly on Globalo.”
- Implementation: Pixel and LinkedIn tag loaded only if “marketing” consent; no extra server logic.

## 5. CONTENT MARKETING (BLOG + AI)

### 5.1 Blog module
- **Tables**: `blog_posts` (id, slug, title, meta_description, body, author_id, published_at, created_at), `blog_categories` (optional).
- **Routes**: `/blog`, `/blog/{slug}`. Controller: `BlogController`. List + single post with SEO meta and schema Article.

### 5.2 AI-generated articles
- Optional: Admin “Generate post” with target keyword → call OpenAI (or similar) → draft saved in `blog_posts` for edit/publish.
- Service: `App\Services\BlogSeoService::generateDraft($keyword)`.

### 5.3 Internal linking
- In post body or via “Related posts” / “Experts in this topic” blocks linking to expert profiles and category pages.

## 6. REFERRAL SYSTEM

### 6.1 Schema
- **Table** `parrainages`: id, parrain_id (user who invites), filleul_id (invited user, NULL until signup), code (unique, e.g. UUID), email_invite (optional), statut (envoye, inscrit, recompense_parrain, recompense_filleul), created_at, inscrit_at.
- **Table** `credits_referral` or reuse `portefeuilles`: reward = credit on wallet (e.g. 500 FCFA each).

### 6.2 Flow
- User gets “Referral” link: `{BASE_URL}/auth/inscription?ref=CODE`. On signup, `filleul_id` and `statut=inscrit` set; after validation (e.g. email verified or first booking), grant credits to parrain and filleul; set recompense_*.
- **Admin**: Config “Referral reward amount” and “Reward after” (e.g. first payment or email verified).

## 8. EMAIL AUTOMATION

### 8.1 Campaign types
| Campaign           | Trigger                    | Content (summary)                |
|--------------------|----------------------------|----------------------------------|
| Welcome            | After registration         | Welcome, verify email, first steps |
| Expert onboarding  | Expert role approved       | How to complete profile, get first mission |
| Abandoned signup   | Inscription started, no confirm (e.g. 24h) | Reminder to confirm email |
| Booking confirmation| Reservation accepted + paid | Date, expert, link to session    |
| Newsletter         | Opt-in, scheduled          | Tips, new experts, blog posts    |

### 8.2 Implementation
- **Config**: SMTP or SendGrid/Mailjet API keys in config/params.
- **Queue table** `email_queue`: id, to_email, template_key, payload JSON, status, sent_at, created_at.
- **Service** `EmailService::queue($templateKey, $to, $payload)`. Cron or background worker sends from queue.
- **Templates**: Stored as PHP/HTML in `app/Views/emails/` or in DB (admin-editable). Placeholders: {{name}}, {{link}}, etc.

## 9. ADMIN GROWTH DASHBOARD

### 9.1 Metrics (sources)
- **Traffic**: GA4 API or manual “Source” report (or embed GA dashboard).
- **Conversions**: From GA4 (sign_up, purchase) and from DB (count inscriptions, paiements).
- **Referrals**: From `parrainages` (count by statut, total rewards given).
- **SEO**: Manual or third-party (e.g. Search Console API) — “Top queries”, “Average position”.

### 9.2 Dashboard UI
- Blocks: Traffic (sessions, source breakdown), Conversion funnel (visit → signup → booking → payment), Referral stats (invites, signups, rewards), Recent campaigns (email open/click if tracked), SEO summary (optional).
- Data: Fetched from DB (referrals, conversions) and from config (GA view ID) for embedding or API.

## 10. PERFORMANCE & SECURITY

- **Fast load**: Async/defer scripts, preconnect, minimal blocking CSS.
- **Scalable**: Stateless app; DB connection pooling; queue for emails; cache sitemap for 1h if needed.
- **Secure tracking**: No PII in GA/Pixel event params; consent before any third-party script; HTTPS only for production.

## Marketing automation workflow (summary)

1. **Welcome email** — Trigger: user created. Action: queue email with template `welcome`, placeholders `{{name}}`, `{{verification_link}}`.
2. **Expert onboarding** — Trigger: expert `valide_par_admin = 1`. Action: queue email `expert_onboarding` with link to complete profile.
3. **Abandoned signup** — Cron daily: find users with `email_verifie = 0` and `created_at` &lt; 24h ago, no email sent in last 24h; queue `abandoned_signup_reminder`.
4. **Booking confirmation** — Trigger: after `PaymentService::processPaiementEscrow` success. Action: queue `booking_confirmation` to client with reservation details.
5. **Newsletter** — Manual or scheduled: select opted-in users, queue `newsletter` with content from admin or template.

Implementation: store templates in DB or files; `EmailService::queue($template, $to, $payload)`; worker or cron sends via SMTP/SendGrid.
